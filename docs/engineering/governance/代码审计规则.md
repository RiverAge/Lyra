# 代码审计规则

> 适用范围:通用(审计纪律,与具体技术栈无关)。示例中的 Python/FastAPI 术语可按栈替换。
> 来源:从 camping 项目 `代码审计规则.md` 改写,适配 Lyra(Python FastAPI + Vue3,自用单用户)。删除了资金链路、多实例一致性等 camping 业务特定章节。

---

## 1. 目标

1. 统一代码评审与交付审计的底线规则
2. 避免 token / session / ownership 类问题反复出现
3. 审计判断优先面向长期终态架构与稳定演进,不以「兼容当前代码形态」或「短期快速落地」为默认最优解
4. 本文档只保留需要人工判断的审计规则;已由脚本/lint/CI 强制执行的约束,只保留摘要与引用

---

## 2. 审计入口

先判定改动类型,再决定优先检查哪些规则,不按章节顺序平均用力。

### 2.1 常见改动类型

1. 资源 ownership / 认证 / token 策略
2. 跨切面机制替换(FastAPI 中间件、依赖项、装饰器、统一封装)
3. 结构重构且要求不改变原有行为
4. 异步任务 / SSE / WS / 状态推进
5. 文档 / 测试 / 构建治理

### 2.2 优先检查顺序

1. 先找最危险的行为断点,再补完整性检查
2. 对高风险域(ownership、token、跨切面替换),优先确认是否存在 fail-open、越权、状态错乱、测试失真
3. 若问题具有可重复扩散的结构性特征,应额外记录规则化机会(guard / lint / 测试模板)

## 3. 强制规则

### 3.0 终态优先原则

1. 审计结论优先评估「长期正确性、领域建模完整性、架构一致性」,不得默认以最小改动或短期兼容作为通过依据
2. 对同一需求若存在「临时兼容方案」与「终态建模方案」,默认以终态方案为推荐与评判基准;临时方案仅可作为明确过渡阶段,并需附退场计划
3. 审计报告必须显式区分:过渡性修复(Tactical Fix);终态方案(Strategic End-state)
4. 若交付选择过渡性修复,必须同时给出:未采用终态方案的原因;终态方案目标版本;差距清单与收敛时间
5. 对 token、ownership、核心数据模型等高价值域,禁止长期依赖「推导字段替代领域字段」「展示层补语义」等做法;应优先固化为可追溯的数据契约

### 3.1 本地存储约束

1. 严禁在业务代码中随意新增敏感本地持久化项(token、业务详情、扫描状态)
2. 若确需新增,必须有明确评审记录、生命周期与失效策略;未完成评审的改动审计结论判定为不通过

### 3.2 业务变更收口与清理

1. 涉及业务口径、状态机、流程或 ownership 变更时,必须同步检查并清理无关代码与冗余实现
2. 涉及数据模型变更时,必须同步评估并处理无效字段、废弃字段与无用查询链路
3. 文档、接口、数据库结构三者口径必须一致;禁止只改其一导致「半收口」状态
4. 对无法立即清理的历史项,必须在交付报告中列出:未清理项清单;影响范围;后续清理计划与责任人

### 3.3 初始化脚本边界

1. Alembic baseline migration 仅用于「从零初始化」,不得混入增量迁移逻辑
2. 初始化脚本中禁止出现面向存量库兼容的动态 DDL 片段或迁移式写法
3. 初始化所需的索引、约束、默认值应直接体现在 `CREATE TABLE` 定义中
4. 如需兼容存量环境,必须使用独立迁移脚本(与 baseline 分离)
5. 审计发现「初始化 + 迁移混写」时至少判定 P1;若导致初始化口径不确定或回放不一致,判定 P0

### 3.4 业务 ID 精度安全

1. 前后端接口中的业务主键与关联 ID 必须按字符串契约传递与处理
2. 该约束由 quality-gate 强制执行(若后续建 id-guard);审计时重点关注 bypass/ignore、新增边界是否未被 guard 覆盖,以及是否已造成主流程错误

> Lyra 单用户自用,ID 精度风险低于 camping 多用户场景,但 SQLite rowid 是 int64,JSON 序列化到前端时仍需注意。建议用字符串传输。

### 3.5 反过度设计约束

1. 终态优先不等于一次性堆叠复杂度;方案必须遵循「最小可验证闭环(MVC)」原则,以最少必要变更达成核心目标
2. 禁止将「可选扩展点」与「当前必需能力」强绑定交付;可延后能力应以清晰扩展点保留,不得在本迭代提前落地完整实现
3. 新增状态、字段、模块、任务链路前,必须给出必要性证明:不新增会导致的具体缺陷;新增后的可观测收益;更小方案为何不足
4. 审计应优先拒绝「为未来假设场景预埋大量实现」的改动
5. 若同一目标存在「最小闭环方案」与「全量理想化方案」,默认先验收最小闭环方案;全量方案需拆分里程碑并定义验收边界
6. 发现过度设计时至少判定 P1;若引入高风险复杂度(状态错乱、回滚不可控),判定 P0

> Lyra 自用项目尤其需要这条防膨胀。

### 3.6 状态落库约束(从多实例约束退化,保留核心)

1. 状态(扫描进度、token 缓存、credits 队列)不依赖进程内内存作为唯一真源——重启不能丢
2. 推送链路(SSE)的断线重连不能依赖进程内存,必须有可恢复的真源(如 SQLite 查 `scanner_status` 表)
3. 若推送端点使用 Query Token,必须在设计与实现中给出泄露风险控制方案与有效期策略

> Lyra 单实例自用,多实例一致性约束不适用,但「状态落库不依赖进程内存」对单实例也成立(容器重启)。

### 3.7 测试同步

1. 涉及业务口径、ownership、状态机、流程或数据模型变更时,必须同步新增或修改对应的测试用例
2. 若变更将某个校验/守卫从一层移动到另一层(如路由函数体 → FastAPI 依赖项),测试的 `TestClient` 调用链必须覆盖新的执行点,禁止出现「测试绕过新校验层」的假通过
3. 测试环境与生产环境的校验链路必须口径一致。禁止生产有依赖项但测试用 standalone 绕过
4. 对无法立即补齐测试的历史遗留模块,必须在交付报告中列出缺口清单与补齐计划

### 3.8 审计方法约束(重点)

1. 审计结论必须基于实际代码读取,不得仅凭变更摘要、commit message 或 agent 声明的输出来判断
2. 每个变更文件必须逐文件审查,确认实际代码与变更描述一致
3. 间接影响的文件(如测试、配置、依赖项注册)必须纳入审查范围,不得只审查变更描述中明确列出的文件
4. 审计发现若依赖于「agent/工具声称已完成」但未通过代码阅读验证,该审计结论无效
5. 审计必须包含可执行验证:至少运行 `ruff check`、`mypy` 与相关测试套件(`pytest`),确认无回归。不得仅以「通过」的口头声明作为验证依据
6. 跨切面变更(删除/替换 FastAPI 中间件、依赖项、装饰器等全局生效机制)必须执行全量覆盖审计:列出受影响范围内所有端点或目标,逐一确认替代机制已覆盖。具体步骤:
   1. 枚举受影响的完整端点/目标清单(如 `grep` 所有路由函数)
   2. 枚举替代机制的覆盖清单(如所有 `Depends(verify_ownership)`)
   3. 做差集 = 缺口清单
   4. 缺口逐项处理或标记为有意豁免

### 3.9 行为等价性审计

1. 若任务类型为重构、下沉、统一封装、守卫替换或其他宣称「不改变原有行为」的结构调整,审计必须显式验证行为等价性
2. 审计至少回答:原行为是什么;新行为是否逐项等价;是否存在从 fail-close 变成 fail-open 的路径;是否存在旧行为未被测试锁住导致的伪通过
3. 若无法证明行为等价,应按真实行为变化处理,不得继续按「纯重构」验收

## 4. 高风险专题

### 4.1 查询性能风险

列表/分页/批处理接口禁止业务循环中逐条关联查询(N+1);关联数据优先批量查询(`IN(...)`/`JOIN`)后内存映射回填;PR/交付说明必须写明查询次数上界;受历史架构限制暂无法消除 N+1 必须标注影响范围、峰值风险、治理计划。

### 4.2 机器规则引用

- 时间契约:若后续建 time-guard(强制 epochMillis 或统一 datetime 入口),审计关注 bypass/ignore
- ID 契约:若后续建 id-guard,审计关注 bypass/ignore
- 前端 disable/bypass:由 `scripts/quality-gate/check-disable.mjs` + `eslint.config.js` 强制
- 审计关注 bypass/ignore、guard 未覆盖新边界、绕开 guard 预期约束
- guard 通过不等于审计通过;反复出现同类问题应转为新 guard/lint/测试基线

> Lyra 后端 guard 暂缓(见 `AGENTS.md` §2.1),当前只有前端 machine-enforced 规则。

## 5. 审计分级

### 5.1 P0(阻断合并)

1. 造成 ownership 越权风险的缺陷(单用户场景下,指能操作到不属于当前会话资源的数据)
2. 未经评审新增敏感本地持久化(token、业务数据)且存在泄露或串用风险
3. 业务口径变更后出现严重不一致,导致主流程不可用或结果错误
4. 状态落库缺失导致容器重启后进度丢失、扫描状态错乱

处理要求:必须先修复再合并;审计报告给出复现方式与修复证据。

### 5.2 P1(原则上修复后合并)

1. 不直接阻断,但造成明显业务偏差、数据歧义或持续运维成本上升
2. 业务改动后存在明确冗余代码/字段未清理,且已影响阅读、联调或后续迭代
3. 文档、接口、数据库口径有差异,短期可能引发误用
4. 以临时兼容方案替代终态方案,且未给出明确收敛路径与时间承诺
5. 改动引入可重复扩散的结构性回归口子,即使当前影响面有限,原则上至少判定 P1

处理要求:同迭代内完成修复,或提供明确延期计划并获负责人确认;审计报告记录影响范围和回收计划。

### 5.3 P2(建议优化)

1. 不影响当前正确性,但会降低可维护性、可读性或一致性
2. 可在后续小版本中安排治理
3. 已选择过渡性方案且终态方案、版本与责任人已明确,仅剩非阻断性的工程化收敛工作

处理要求:可带问题合并,但必须登记待办项。

## 6. 交付模板(审计必填)

1. **改动文件清单**(绝对路径 + 变更类型)
2. **需求点与实现映射**(逐条对应)
3. **接口/数据结构前后对比**(如有变更)
4. **验证结果**:`ruff check`;`mypy`;关键 `pytest` 用例;回归检查;机器规则结果摘要(如 style-guard、check-disable)
5. **清理清单**:已清理项;未清理项(含原因)
6. **风险与结论**:P0/P1/P2 分级;是否允许合并;后续行动计划(如有)
7. **规则化机会**:本次问题是否适合沉淀为 guard/lint/测试模板;若适合,建议归属到哪个机制

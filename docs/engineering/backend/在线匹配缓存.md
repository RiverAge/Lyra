# 在线匹配缓存(payload + candidate 两层)

> 适用范围:Python 后端。Lyra 在线歌词匹配的跨请求磁盘缓存设计(两层)。
> 代码真源:`backend/lyrics/lyric_match/payload_cache.py`(raw payload 层) +
> `backend/lyrics/lyric_match/candidate_cache.py`(search/detail 候选层)。

---

## 0. 一句话结论

**在线匹配的瓶颈是外部 API 网络(8-17 次往返 + 强制 sleep,单次 8-20s),不是 CPU(converter 微秒级)。两层进程级磁盘缓存:

1. **candidate 缓存**(search+detail 层,2026-07 加):key = `provider.source + 查询指纹(title|artist|album|duration_ms|limit)` 的 sha1 前 16 字符,value = 候选列表(含 raw)。砍掉同一首歌第二次点「在线匹配」的全部 search/detail 网络往返(8-17 次)——**这是耗时大头**。
2. **payload 缓存**(fetch_lyrics 层):key = `provider.source + candidate.id`,value = raw payload。砍掉拉词往返 + 候选切回/preview 拉词的重复请求。

两层 TTL 均 7 天,目录挨着 db(`<db_dir>/lyric_cache/` + `cands/` 子目录)。candidate 层是大头——payload 层只省最后的拉词几百毫秒。实测:同一首歌第二次匹配从 ~9s 降到接近秒回(命中 candidate + payload 双缓存,零外部请求)。**

---

## 1. 根因:零跨请求缓存

每次点「在线匹配」完整重跑:
- NetEase search:1+5 次 weapi POST,每次 0.15s sleep = 0.9s 纯等
- QQ search:最多 5 次 GET,每次 0.8s sleep = 4s 纯等
- fetch_lyrics(best + 可能的 QQ 探测):1-4 次网络 + 3DES 解密

总 8-17 次外部往返,8-20s。瓶颈在 sleep + 网络延迟。

**现状零缓存**:
- QQ 的 `self._qrc_cache`(qq.py)是**实例缓存**——provider 每次请求新建(`lyrics_match_routes._build_providers`),请求结束 `close()` 销毁,实例缓存随之丢失。这是「单次请求内防重复、跨请求丢」的现状。
- 后端无内存 dict / 磁盘 / SQLite 表 / Redis 存匹配产物。
- 前端 store 只存单首(切 track 即丢),且只省 HTTP 一跳,省不掉后端网络。

---

## 2. 设计决策(及为何这样选)

| 维度 | 决策 | 理由 |
|---|---|---|
| **缓存层** | 只做后端 payload 缓存 | 瓶颈在后端网络,前端缓存只省 HTTP 一跳,复杂度不值 |
| **粒度** | 缓存 raw payload(非最终 TTML) | converter 是纯 CPU 微秒级,重跑无负担;raw 是最稳的缓存值(TTML 输出随 source 参数变体,key 管理复杂) |
| **介质** | 磁盘 JSON 文件(`.cache/`) | 结构轻、易手动清理、TTL 靠 mtime;非 SQLite(缓存是临时数据,不需结构化查询) |
| **键** | `provider.source + ":" + candidate.id` | provider song id 在 provider 侧永久稳定(网易/QQ 歌曲 id)。track_id(rowid)同一首歌不同 rip 不命中;query 归一化易撞车 |
| **TTL** | 7 天 | provider 侧数据会更新(网易改词、QQ 补注音),不能永久。mtime 过期不主动删,下次 set 时 replace 覆盖 |

---

## 3. 缓存语义:尊重 provider 自带判断(关键)

`fetch_lyrics_cached` 不是无脑缓存——它**尊重各 provider 自己的语义判断**,决定哪些 payload 值得缓存:

### QQ payload(`qq.py:fetch_lyrics` 四个返回分支)

| payload 类型 | `_qrc_status` | 缓存? | 理由 |
|---|---|---|---|
| 成功 | (无) | ✓ | 真实歌词,可缓存 |
| placeholder | `placeholder` | ✓ | QQ 判定这歌真没词,缓存省重复探测 |
| no_content | `no_content` | ✓ | 无加密内容,缓存省重复探测 |
| **解密错误** | `decrypt_error: ...` | **✗ 绝不** | 对齐 qq.py:303 注释「Don't cache decrypt errors — allow a retry」——允许后续重试命中真实数据 |

> **双保险**:`payload_cache.set` 内部 `status.startswith("decrypt_error")` 跳过;QQ fetch_lyrics 的 decrypt_error 分支本身也不写实例缓存。两层都跳,语义对齐。

### NetEase payload
无 `_qrc_status` 字段,直接缓存 raw dict(含 yrc/lrc/tlyric/romalrc)。空/placeholder 由消费侧 `lyric_payload_has_text` 判,缓存层不操心——`payload_cache.set` 收到 None 直接 return(None 不落盘,下次回源)。

---

## 4. 注入点:包 provider 层,覆盖三处 fetch_lyrics

`fetch_lyrics` 有四个调用点,只有包在 **provider 层**才能全覆盖:

| 调用点 | 文件 | 覆盖方式 |
|---|---|---|
| runner 取 best 候选词 | `runner.py:_fetch_best_with_qq_fallback` | `fetch_lyrics_cached` helper |
| runner QQ 回退 NetEase | `runner.py` netease_line 分支 | `fetch_lyrics_cached` helper |
| preview 端点按 candidate.id 取词 | `lyrics_match_routes.py` | `fetch_lyrics_cached` helper |
| QQ `find_qrc_candidate` 内部探测 | `qq.py` | **改 qq.py fetch_lyrics 本身**(开头查跨请求缓存,末尾各落盘点 set) |

QQ 内部不能靠 helper——`find_qrc_candidate` 调的是 `self.fetch_lyrics`,helper 包不到。所以 qq.py 的 fetch_lyrics 自己在开头查跨请求缓存(命中喂给实例缓存)、末尾各 payload 落盘点 `await get_payload_cache().set(...)`,让 find_qrc_candidate 自动吃跨请求缓存。

NetEase 无内部探测逻辑,靠 runner/routes 的 `fetch_lyrics_cached` 覆盖即可。

---

## 5. 目录位置:`<db_dir>/lyric_cache/`

挨着 SQLite db,零新配置。用 `config.db_path_resolved()` 拿 db 绝对路径,`.parent / "lyric_cache"`。

**为何不挂别处**:
- 不挂 `<music_library>/.lyrics/`:那是**用户数据**(采纳的 sidecar TTML),缓存是 Lyra 内部临时数据,分开
- 不加 `LYRA_CACHE_DIR` 配置字段:本次零新配置最省;未来要可配再加(跟 `log_dir` 同模式)
- 不挂 db 内 SQLite 表:缓存是临时数据,不需要结构化查询,JSON 文件够轻

**手动清理**:删 `lyric_cache/` 目录即可,下次匹配自动重建。

---

## 6. 并发与原子性

- **单 worker 防击穿**:`payload_cache._LOCK`(asyncio.Lock)保证同 key 并发 miss 时第二个等第一个写完再读,不都回源。uvicorn 默认单 worker,fetch_lyrics 是 await,并发场景真实存在。
- **原子写**:`tmp.write_text` → `tmp.replace(p)`,避免并发读到半截 JSON。读只认 `.json`,残留 `.tmp` 不影响。
- **损坏自愈**:读失败(解析错/文件损坏)→ 返回 None 回源,不让匹配崩。日志 warning。

---

## 7. 为何不缓存这些

| 不缓存 | 理由 |
|---|---|
| 最终 TTML | converter 微秒级重跑无负担;TTML 输出随 source 参数变体,key 管理复杂;raw 是最稳缓存值 |
| ~~search 结果(候选列表)~~ | **已加 candidate 缓存**(见 §10)。原顾虑「query 归一化撞车」改用 sha1(title|artist|album|duration_ms|limit) 规避;实测 search+detail 是耗时大头,值得缓存 |
| scoring/decision 结果 | 是纯函数,算一遍微秒级,无网络,不需要缓存(candidate 缓存命中后照常 score_candidate 重算,幂等) |
| track_id 级整匹配结果 | 同一首歌不同 rip 格式 track_id 不同不命中;且 provider song id / 查询指纹粒度更准 |

## 10. candidate 缓存层(search+detail,2026-07-16 加)

### 起因

payload 缓存上线后发现「第二次点同歌还是一样慢」。实测日志拆解(同一首《人生》两次匹配):
- 第一次 10.3s:6 次 netease cloudsearch + 1 次 song/detail + 5 次 QQ client_search_cp + 1 次 QQ lyric_download
- 第二次 8.8s:6 次 cloudsearch + 1 次 detail + 5 次 client_search_cp(**全重跑**)+ **0 次 lyric_download**(payload 缓存命中了)

payload 缓存只省掉了最后的拉词(lyric_download),~9s 的耗时大头是 search/detail 那 12+ 次网络往返,缓存没盖到。

### 设计

- **层**:`resolve_candidates` 内,给每个 provider 的 `search(q, limit)` + `detail(cands)` 序列套缓存。**缓存整个 search+detail 合并结果**(不只是 search),把 netease 的 song/detail 也一并省掉(QQ detail 是 no-op)。
- **键**:`provider.source + ":" + sha1(title|artist|album|duration_ms|limit)[:16]`。
  - 用 q 本身的四字段 + limit 做指纹,不覆盖 `make_queries` 的搜索词变体——变体是 provider 内部细节,同 q 永远生成同一组变体,指纹用 q 即可。
  - **duration 纳入指纹**:netease search_match 用 duration 筛选;同一首歌不同 rip 的 duration 略不同 → 不命中是正确的(duration 影响搜索结果)。duration 取整毫秒抹掉浮点尾数抖动。
  - 用 sha1 而非原文:搜索词含中文/空格/特殊字符,文件名 sanitize 麻烦;sha1 hex 文件名安全且定长。
- **值**:detail 后的 `list[Candidate]`,用 `dataclasses.asdict` 序列化(含 `raw` 字段——score_candidate 读 `c.raw.get("artists")`,round-trip 不能丢),读回 `Candidate(**_filter_candidate_fields(item))` 重建。
- **空列表也缓存**:「搜不到」是稳定结果,缓存省掉对「这歌搜不到」的反复探测。
- **异常路径不缓存**:provider.search/detail 抛异常 → `resolve_candidates` 的 try/except 跳过该 provider,`cache.set` 在异常路径不执行,允许下次重试。
- **TTL/介质/并发**:复用 payload_cache 同款——7 天 TTL(mtime)、`<db_dir>/lyric_cache/cands/` 子目录、asyncio.Lock 防击穿、tmp+replace 原子写、损坏自愈回源。

### 重建 Candidate 的字段过滤

`_filter_candidate_fields` 显式白名单 Candidate 字段:防御性处理未来 Candidate 加/减字段时旧缓存的 graceful 退化(多余字段丢弃,缺失字段走 dataclass 默认值),避免 `Candidate(**d)` 因多余字段 TypeError 崩匹配。

### 零决策回归

candidate 缓存命中后,`resolve_candidates` 照常 `score_candidate(q, c)` 重新打分(打分是纯 CPU 幂等,缓存只存 search/detail 的原始字段,score/reasons/penalties 会被重算覆盖)。decision/scoring 逻辑零改动——缓存纯卡在 search/detail 网络层,之下(fetch_lyrics → payload 缓存)和之上(scoring/decision)都不动。
